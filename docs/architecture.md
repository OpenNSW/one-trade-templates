# Architecture

## What this repo is

A language-agnostic JSON data repository. It ships workflow and task templates for the OneTrade platform. There is no application code — only:

- `templates/` — the JSON templates themselves (93 files across 4 workflow groups).
- `manifest.json` — generated index of every template, used by consumers.
- `scripts/` — Node.js tooling for validation and manifest generation.
- `docs/` — documentation (this folder).
- CI configuration in `.github/workflows/`.

Any project that can parse JSON can consume this repo. See [`consumption.md`](consumption.md) for how.

## Why no code

The templates are pure data. Any language that can read JSON — Node, Python, Go, browser, mobile — consumes them identically. Keeping the repo language-agnostic means downstream projects don't inherit a runtime or package-manager dependency just to read configuration.

## File tree

```
templates/
├── <group>/
│   ├── <group>_workflow.json          # top-level workflow definition for the group
│   └── N-step-name/                   # one directory per step; number determines order
│       ├── workflow.json              # the step's sub-workflow definition
│       ├── <task>.json                # task template (e.g. userinput.json, payment.json)
│       └── <task>_jsonform.json       # JSON Schema form definition paired with the task
```

The four current groups are `cda`, `customs`, `fcau`, `npqs`. See [`workflows.md`](workflows.md) for what each one models.

## How files reference each other

Every template file has a unique top-level `id` field (see [`conventions.md`](conventions.md)). Workflow definitions reference other templates by id via the `task_template_id` field on workflow nodes.

There are two levels of reference:

1. **Top-level workflow → step workflows.** The top-level `<group>_workflow.json` defines the workflow as a sequence of `TASK` nodes. Each `TASK` node's `task_template_id` points to the step's `workflow.json` `id`.

   ```jsonc
   // templates/npqs/npqs_workflow.json
   {
     "id": "npqs-export-phytosanitary-reg",
     "nodes": [
       { "type": "TASK", "task_template_id": "npqs-apply-phyto-cert-flow" /* → templates/npqs/1-application/workflow.json */ },
       { "type": "TASK", "task_template_id": "npqs-wait-sample-received-flow" /* → templates/npqs/2-wait_on_sample/workflow.json */ }
     ]
   }
   ```

2. **Step workflow → task definitions.** A step's `workflow.json` orchestrates the actual task templates in the same directory. Each `TASK` node points to a sibling file by id.

   ```jsonc
   // templates/npqs/1-application/workflow.json
   {
     "id": "npqs-apply-phyto-cert-flow",
     "nodes": [
       { "type": "TASK", "task_template_id": "npqs-apply-phyto-cert--user-input" /* → userinput.json */ },
       { "type": "TASK", "task_template_id": "npqs-apply-phyto-cert--officer-review" /* → reviewerinput.json */ }
     ]
   }
   ```

## `task_type` is not a reference

The `task_type` field (appears on task definition files) holds an engine primitive enum value like `PAYMENT`, `WAIT_FOR_EVENT`, `SIMPLE_FORM`, `HTTP_POST`. These are not template ids — they tell the workflow engine *how* to execute the task. Two unrelated templates can share a `task_type` value; they cannot share an `id`.

## The `manifest.json` entry point

Generated by `scripts/generate-manifest.mjs`. Contains two indexes:

- `workflows[]` — structured tree, ordered: top-level workflow → numbered steps → files.
- `byId{}` — flat map from `id` to file path. The 1:1 lookup that consumers use.

See [`manifest.md`](manifest.md) for the full schema.

## Where the rules live in code

- **`scripts/validate-templates.mjs`** — enforces top-level-object, `id` present, `id` globally unique.
- **`scripts/generate-manifest.mjs`** — walks `templates/` and writes `manifest.json`.

CI runs Prettier + the validator + a manifest-sync check. See [`lint-rules.md`](lint-rules.md).
